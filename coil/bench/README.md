# Benchmarks — Coil vs C

Compute-bound microbenchmarks, each written twice (`X.coil` and `X.c`) with the
**same algorithm**, to compare Coil's compiled output against C through the same
LLVM backend.

```sh
bench/run.sh             # build + verify + time every benchmark -> bench/RESULTS.md
bench/run.sh fib loop    # a subset
```

`run.sh` builds each program both ways and **checks they print the same result
before timing** (a fast wrong answer is meaningless):

- **Coil -O3** — `coil build` (the LLVM `-O3` pipeline Coil runs on every object).
- **C -O3** — the fair baseline: same LLVM backend, same optimization tier.

Timing is [hyperfine] (`brew install hyperfine`).

## The benchmarks

| file | what it stresses |
|---|---|
| `fib` | naive recursive Fibonacci — function-call + integer-add overhead |
| `tak` | Takeuchi — deeply nested non-tail recursion |
| `loop` | a tight integer multiply-accumulate loop |
| `float` | `double` divide+add throughput (harmonic sum) |
| `memory` | fill + sum a 20M-element heap array (load/store/GEP + bandwidth) |

Coil has no loop keyword — iteration is **tail recursion**, which the optimizer
turns into a real loop (tail-call elimination), so `loop`/`float`/`memory` don't
recurse on the stack.

## Result (Apple M-series, arm64)

Coil is at **parity with C `-O3`** — every benchmark lands within ~5% (both
directions), i.e. measurement noise:

| benchmark | Coil vs C -O3 |
|---|---:|
| fib    | 1.01× slower |
| tak    | 1.02× slower |
| loop   | 1.00× (even) |
| float  | 1.01× faster |
| memory | 1.05× faster |

This is the expected outcome: Coil emits straightforward LLVM IR and runs the
same `-O3` pipeline clang does (mem2reg, inlining, GVN, loop opts, tail-call
elimination), through the same backend — so the machine code is in the same
league. See `RESULTS.md` for the full hyperfine tables (regenerated by `run.sh`).

[hyperfine]: https://github.com/sharkdp/hyperfine
