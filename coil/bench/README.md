# Benchmarks — Coil vs C vs Zig

Two questions these benchmarks answer:

1. **Does Coil's *generated code* keep up with C and Zig?** Each program is written
   three ways (`X.coil`, `X.c`, `X.zig`) with the **same algorithm**, built at the
   highest optimization tier, verified to print the same result, then timed.
2. **Does the *compiler* scale?** `compile_scale.sh` charts compiler wall-time as a
   program grows, to measure the whole-program-monomorphization + comptime-macro risk.

```sh
bench/run.sh                  # build + verify + time every benchmark -> bench/RESULTS.md
bench/run.sh fib slicesum     # a subset
bench/compile_scale.sh        # compile-speed at scale     -> bench/COMPILE_SCALE.md
```

`run.sh` builds each program every way and **checks they print the same result
before timing** (a fast wrong answer is meaningless):

- **Coil -O3** — `coil build` (the LLVM `-O3` pipeline Coil runs on every object).
- **C -O3** — `cc -O3`: same LLVM backend, same optimization tier.
- **Zig** — `-OReleaseFast`. Optional: skipped if `zig` is absent or can't link on
  the host. (On macOS 26 the bundled `zig` linker can't resolve the newer libSystem,
  so the harness compiles Zig to an object and links it with the system `cc`.)

Timing is [hyperfine] (`brew install hyperfine`).

## The benchmarks

**Compute micros** — does the backend keep up on raw arithmetic/memory:

| file | what it stresses |
|---|---|
| `fib` | naive recursive Fibonacci — function-call + integer-add overhead |
| `tak` | Takeuchi — deeply nested non-tail recursion |
| `loop` | a tight integer multiply-accumulate loop |
| `float` | `double` divide+add throughput (harmonic sum) |
| `memory` | fill + sum a 20M-element heap array (load/store/GEP + bandwidth) |

**Abstraction costs** — do Coil's *language features* compile away, or do they cost?
These are the interesting ones: a frontend can leave performance on the table here in
a way the scalar micros never reveal.

| file | what it stresses |
|---|---|
| `structcall` | a 3-field struct passed **by value** to a hot leaf fn (AAPCS64/SysV coercion) |
| `slicesum` | a cache-resident `(slice i64)` indexed 200M times via `slice-get` (fat pointer + a generic accessor) vs C's raw `a[i&1023]` |
| `genreduce` | a **monomorphized generic** `gsum[T]` at `i64` vs a hand-written i64 loop |

Coil has no loop keyword — iteration is **tail recursion**, which the optimizer turns
into a real loop (tail-call elimination), so the loops don't recurse on the stack.

## Result (Apple M-series, arm64)

Coil is at **parity with C `-O3` and Zig `-OReleaseFast`** on every benchmark —
including the abstraction ones, which is the point: **struct-by-value, slices, and
monomorphized generics compile away to the same machine code C/Zig produce.** See
`RESULTS.md` for the full hyperfine tables (regenerated by `run.sh`); the abstraction
benchmarks have tight variance (±~1%) and land even.

This is the expected outcome: Coil emits straightforward LLVM IR and runs the same
`-O3` pipeline clang does (mem2reg, inlining, GVN, loop opts, tail-call elimination),
through the same backend, and its abstractions lower to that same IR.

## Compile speed at scale (`compile_scale.sh`)

The honest risk (docs/FUTURE_WORK.md §12): whole-program monomorphization + a comptime
macro/elaboration pass could blow up super-linearly on large programs. `compile_scale.sh`
generates programs of growing size (a reachable chain of functions, 10% generic at two
types, 20% using the `when` macro) and separates the **front end** (`emit-ir`:
read→expand→check→mono→IR, no optimizer) from a full **build** (adds LLVM -O3 + link).
The Coil compiler is built `--release` for this.

Measured up to ~8800 LOC, the front end scales **linearly** (~0.03 ms/function once
process startup amortizes) — no super-linear blowup from monomorphization or macros.
The dominant cost at scale is the LLVM `-O3` backend, shared with clang/Zig. See
`COMPILE_SCALE.md` (regenerated by `compile_scale.sh`).

## Compiler throughput vs clang & zig (`compile_compare.sh`)

`compile_scale.sh` measures Coil alone; `compile_compare.sh` puts it next to the other
toolchains. It generates the **same** program (a chain of plain integer functions, no
generics/macros — the shared subset) in Coil, C, and Zig, and times each compiler
producing an executable in **both** build modes. The Coil compiler is built `--release`
to match the release clang/zig binaries; Zig needs **≥ 0.16** to link on macOS 26 (see
the §"Zig" note above) and is skipped otherwise. Results in `COMPILE_COMPARE.md`.

What it shows (≈4000 functions, this host):

- **Release** (`-O3` / `-OReleaseFast`): Coil is **~1.15× clang and faster than zig** —
  essentially at parity. (It was ~2.4× *slower* than clang until functions were given
  `internal` linkage; see below.)
- **Debug** (`-O0` / Zig Debug / Coil `-g`): Coil is **competitive with clang `-g -O0`**
  and **far faster than Zig's Debug mode**. Coil's lack of a true Debug/Release build
  *mode* split (its only non-O3 path is `-g`) is the main caveat to close here.

### Why it was slow — and the fix (internal linkage)

The breakdown (`COIL_TIME=1 coil build …` prints per-phase timings) showed Coil's own
compiler is fast (~75 ms for 4000 functions: read→expand→check→mono→IR); the entire
compile-time gap was the LLVM `-O3` pipeline + machine-code emission. The cause was
**not** messy IR (Coil emits clean SSA) — it was **linkage**: Coil emitted every
function with *external* linkage, so the optimizer could never delete a function it
inlined or proved dead. On a deep call graph that bloats the code O(n²); in general it
means optimizing and emitting machine code for far more than survives. Giving private
functions **`internal` linkage** (like C's `static`) — done for executables, which are
detected by defining `main` — let LLVM prune, and object emission for the 4000-function
chain dropped **355 ms → 5 ms**. A library or freestanding image (no `main`) keeps
external linkage until an explicit symbol-export feature lands (docs/SYMBOL_EXPORT.md).
