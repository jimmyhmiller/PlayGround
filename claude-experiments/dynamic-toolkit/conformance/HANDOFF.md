## 1. The conformance harness (already built — use this as your scoreboard)

Files in `conformance/`:

| file | what it is |
|------|------------|
| `corpus.edn` | one Clojure expression per line (50 to start; add more) |
| `oracle.tsv` | expected `pr-str` output, generated by real Clojure |
| `run.sh` | runs every expr through `clojure-jvm`, restarting past crashes, into `/tmp/ours.tsv` |
| `tests/oracle_runner.rs` (in `crates/clojure-jvm/tests/`) | the our-side runner (env-driven: `CORPUS`, `RESULTS`, `START`) |

### Regenerate the oracle (do this whenever you add to the corpus)

The project note says: **always run the `clojure` CLI from `/tmp`** (running
it in the repo picks up our `core.clj` and confuses it).

```bash
cp conformance/corpus.edn /tmp/corpus.edn
cat > /tmp/oracle.clj <<'EOF'
(let [lines (clojure.string/split-lines (slurp "/tmp/corpus.edn"))]
  (doseq [[i line] (map-indexed vector lines)]
    (when (seq (clojure.string/trim line))
      (let [out (try (pr-str (eval (read-string line)))
                     (catch Throwable t (str "ERROR:" (.getMessage t))))]
        (println (str i "\t" (clojure.string/replace out #"\n" "\\\\n")))))))
EOF
cd /tmp && clojure -M /tmp/oracle.clj 2>/dev/null > /tmp/oracle_out.txt
cp /tmp/oracle_out.txt "$REPO/conformance/oracle.tsv"
```

### Run ours + diff

```bash
./conformance/run.sh            # ~minutes; reloads core per crash (slow but robust)
# then diff /tmp/ours.tsv vs conformance/oracle.tsv (a ~30-line python script
# was used during development; any join-on-index comparison works)
```

Each our-side result is printed via `runtime::pr_str_bits(bits)` (decodes
heap pointers so collections/strings print as real values, not `#<host>`).

## 2. TASK A — Real integers (boxed `Long`)  ← ✅ DONE (5 → 29/50)

> **This task is complete and applied.** The play-by-play below is kept as the
> record of what was done and why (and the gotchas, which still bite). If you're
> here for new work, skip to Task B (§3).

### Why boxed-Long, not tagged fixnums

Real Clojure has a `long` type distinct from `double`. clojure-jvm currently
represents **all** numbers as NaN-boxed `f64` (`NumberExpr::emit` does
`n.as_f64()`; `object_to_nanbox` does `(n as f64).to_bits()`), so every
integer prints/behaves as a double.

The "nice" fix (inline 61-bit fixnums via low-bit tagging) is **deferred** —
it requires rewriting `dynlang`'s value layer **and** `DynGcRuntime`
(hardcoded `NanBoxPolicy`) **and** `DynModule` (`NanBoxTags`), all **shared
by 5 frontends** (beagle, clojure, lox, microlisp, clojure-jvm). That's a
multi-day shared-infra refactor for **zero observable difference** (boxed
Long is observably identical to Clojure: `(class 6)`→Long, `(+ 6 1)`→7,
`count`→3 — only perf differs). So: **boxed Long now, low-bit later** (see §5).

### What's already built and GREEN (staged, currently unused)

All of this is in place and compiles; you're turning it ON, not writing it:

- **`clojure.lang.Long` heap type** — a Raw64 i64 cell. Wired through
  `runtime::HeapTypeIds.long`, `Compiler.long_type_id`, the `obj_type` decl
  in `Compiler::new`, and all ~10 `HeapTypeIds{}` initializers.
- **`crates/clojure-jvm/src/runtime.rs`**: `box_long(i64)->u64`,
  `unbox_long`, `is_boxed_long`; `heap_bits_to_object` decodes the Long
  type_id → `Object::Long`.
- **Numeric-tower externs** (runtime.rs): `cljvm_num_add/sub/mul/div/quot/rem`,
  `cljvm_num_lt/gt/le/ge/equiv`, `cljvm_equals` — they decode operands
  (boxed-Long → Long, else native double), apply the tower, re-encode.
  **Declared** in `Compiler::new` (`Compiler.num: NumExterns`) and
  **registered** in `build_extern_table_for`'s name→ptr `match`.
- **`PendingLiteral::Long(i64)`** + both literal-fill loops (pre-box via
  `box_long`, so integer literals are boxed once in the literal pool).
- **`crates/clojure-jvm/src/lang/value_repr.rs`**: the value codec + the
  numeric tower (`Num{Long,Double}`, `num_add/...`, `num_eq` (`==`) vs
  `val_num_eq` (`=`, type-aware), checked overflow). **7 unit tests, all
  asserted against the 1.11 oracle.** One deliberate deviation: `(/ 7 2)` →
  `3.5`, not the Ratio `7/2` (no rationals yet).

### Why the flip is NOT just "wire arithmetic" (learn from my failed attempt)

I enabled the flip (boxed int literals + `PrimOpExpr` → externs +
`object_to_nanbox` boxing) and it **broke 231/353 tests then hard-aborted.**
Boxed Longs introduce a real heap `type_id` where native-float numbers had
none, so **every type-dispatch site that deals with numbers breaks.** Two
classes found (assume there are more — grep for `type_id` and
`heap_type_ids`):

1. **`Var::bind_root` → `try_object_to_nanbox`** runs during
   `CompilerVars::new`, *before the heap/thread/`heap_type_ids` exist.* You
   **cannot** allocate (box) there. `try_object_to_nanbox` is a
   *non-allocating representability check.*
2. **Protocol dispatch** (`runtime::lookup_protocol_method` /
   `effective_type_id`) hard-panicked: "no impl for method_id on type_id
   <long>". Same will hit `instance?`/`isInstance`, multimethods, `hash`,
   map/set key equality, `(class x)`, and `number?`/`int?`/`integer?`.

### The order to do Task A (keep `cargo test -p clojure-jvm --lib` green at every step)

**Step 0 — type-dispatch pass FIRST (before flipping any encoder).** Make
every number-shape/`type_id` site recognize the `Long` type_id as a number.
Enumerate them: `grep -rn "type_id\|heap_type_ids\|effective_type_id" crates/clojure-jvm/src`.
Known sites to fix:
   - `effective_type_id` / `lookup_protocol_method` (runtime.rs) — route the
     Long type_id to the numeric/`Object` protocol bucket (likely
     `BUILTIN_OBJECT` or a Number builtin).
   - `instance?` / `cljvm_inst_isInstance` — `(instance? Long x)`,
     `(instance? Number x)` must be true for boxed Longs.
   - multimethod dispatch (`(class x)` based) — `(class 6)` → Long.
   - `hash`, and map/set key equality — two `1`s must hash/compare equal
     even as distinct boxes (`cljvm_equals` already handles value equality;
     make hashing consistent).
   - number predicates `number?`/`int?`/`integer?`/`pos?`/`zero?` etc.
   These can land while integers are still floats (they just also accept the
   Long type_id), so the suite stays green.

**Step 1 — `try_object_to_nanbox(Long)` → `None`** (keep it Rust-side as
`Object::Long`; box lazily in the deref path where a thread exists). Verify
`Var` deref of an `Object::Long` root boxes correctly (check the
`deref_bits` / `VarRoot::Object` path).

**Step 2 — flip the encoders (the actual behavior change):**
   - `NumberExpr::emit`: integer literal → `PendingLiteral::Long` →
     `gc_literal` (the code is in a comment there now — uncomment/restore).
   - `PrimOpExpr::emit_value`: route `+ - * /`, unary `-`/`/`, identities
     `(+)`=0 `(*)`=1, and `< > <= >= =` through `c.num.*` externs
     (`=` → `c.num.eq` = `cljvm_equals`). (I had this written; it was
     reverted — see git diff or the memory note for the exact form.)
   - `object_to_nanbox(Long)` → `box_long` (needs a thread; true on all
     literal/heap-store paths).

**Step 3 — sweep integer-returning runtime fns** so results are boxed Longs,
not native floats. Otherwise `(count [1 2 3])` → `3.0`. Find them:
`grep -rn "to_bits()\|nanbox_double\|as f64" crates/clojure-jvm/src/runtime.rs`
— `count`, `nth` index handling, `.length`, `hash`, `RT/nextID`,
`int`/`long` casts, etc. Each `i64` result → `box_long`.

**Step 4 — re-green + measure.** `cargo test -p clojure-jvm --lib`
(update any tests asserting `3.0`-style results — they should now be `3`),
then `./conformance/run.sh`. Expect ~5 → ~30.

### Gotchas
- `box_long` / `object_to_nanbox(Long)` need an **installed mutator thread**.
  Fine on eval/literal-fill paths; NOT fine in early init (that's why
  `try_object_to_nanbox` returns `None` for Long).
- Booleans/nil are unaffected (still NaN-box immediates).
- Overflow: `cljvm_num_*` currently **promotes to double** on long overflow
  (Clojure *throws*) — a documented deviation until exceptions are wired
  through the externs. The corpus never overflows.
- GC: boxed Longs are ordinary heap pointers (`TAG_PTR`); the GC already
  traces them (no inner pointers to scan). No GC change needed.

## 3. TASK B — Lazy sequences (the next big functional gap)

`(range 5)` → nil; `(map …)`/`(filter …)` → empty; `(first (lazy-seq (cons 42 nil)))`
**hard-aborts** (non-unwinding). The lazy-seq realization machinery
(`clojure.lang.LazySeq` thunk forcing + the ISeq/chunked-seq protocols)
doesn't work. Eager structures (vectors, maps, `cons`, `first`/`rest` on
them) DO work. This is the spine of Clojure — tackle after integers.
Start by reproducing `(first (lazy-seq (cons 42 nil)))` and tracing the
abort (it's in the realization/force path).

## 4. TASK C — Full `core.clj` load frontier (lower priority than A/B)

`Session::new_with_clojure_core()` does not complete; it stops partway
(an unresolved-symbol panic around the `when-class`/`case` machinery), so
`reduce` and everything after it never load. Two structural issues to fix
along the way, both noted because they bite any REPL/loader:
- **No compiler error isolation:** the first analyze-time panic leaves the
  `DynModule` with a declared-but-undefined function, so *every* later
  compile panics (`builder.rs:162`). One bad form poisons the session. A
  real Clojure must recover. (Fixing this also makes the conformance harness
  fast — no per-crash core reload.)
- **MOCKED host classes:** `crates/clojure-jvm/clojure/prelude.clj` defs
  ~39 `"#<MOCKED …>"` class stubs that shadow real host classes. `StringBuilder`
  was un-mocked (it's fully supported); others (`Number`, `Long`, exception
  classes, …) still throw via `.isInstance` when exercised. Un-mock each as
  its host support lands. (Don't do the "make host_class win over the Var in
  `analyze_symbol`" shortcut — I tried it; it interns `Class` literals during
  speculative macroexpansion and drifts the literal pool, corrupting later
  `gc_literal` reads. Remove individual mocks instead.)

Run `cargo test -p clojure-jvm --test load_forked_core -- --ignored` to see
how far core load gets (it needs a big stack — spawn the loader on a thread
with e.g. `stack_size(256 * 1024 * 1024)`; clojure.core genuinely needs more
than the 2 MB test-thread default, like JVM `-Xss`).

## 5. DEFERRED — low-bit tagging (perf, not correctness)

Once conformance is good, revisit replacing boxed Longs with inline 61-bit
fixnums (no alloc per int). This is the "right" end-state but is a **shared
`dynlang` refactor**: make `DynFunc` value ops, `DynGcRuntime` (currently
hardcoded `MutatorThread<NanBoxPolicy>` / `collect::<NanBoxPolicy>` /
`JitSafepointSession::<NanBoxPolicy>`), and `DynModule` (`NanBoxTags`)
**generic over `dynvalue::TagScheme`**, with the 4 other frontends keeping
`NanBox` (default) so they don't break. The JIT (`compile_batch<L: TagScheme>`)
+ abstract tag-ops (`Inst::MakeTagged/Payload/IsTag/TagOf`) + the GC's
`LowBitPtrPolicy<3>` are **already** scheme-generic; the tag scheme
(ptr=0, fixnum=1, nil=2, bool=3, char=4) is already locked in
`value_repr.rs`. Do NOT start this until A/B are done.
