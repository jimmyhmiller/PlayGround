# Competitive benchmarks: diffpack vs esbuild, rolldown, rspack, vite

The project claims (1) faster cold builds than every bundler and (2)
substantially lower incremental edit times than every other solution. Until
now those claims were only tested against pinned Rolldown 1.2.0
(`docs/ROLLDOWN_COMPARISON.md`). This document makes them falsifiable: every
number below was measured on this machine by `bench/run.mjs`, and the raw
per-run samples live in `bench/results/results.json`. Nothing here is
estimated. The short version of the verdict, stated up front:

- Claim (1), cold builds, is **not fully supported**: esbuild beats diffpack
  by ~1.5x on the realistic 1,000-module corpus, and rolldown ties diffpack
  on the realistic 10,000-module corpus. Diffpack wins or ties everywhere
  else, but "faster than every bundler on cold builds" is falsified as a
  universal statement.
- Claim (2), incremental rebuilds, **holds** on every measured corpus:
  diffpack's watch rebuild is 2.5-3.4x faster than the best competitor
  (rolldown's experimental incremental mode) and 3-10x faster than the rest —
  under a methodology that is biased *against* diffpack (its number includes
  filesystem-watch detection latency; rolldown's and esbuild's do not).

## Methodology

### Corpora

Both synthetic profiles are generated by `bench/gen.mjs`, adapted directly
from `oracle/benchmark.mjs`'s live-output generator, and share its exact
dependency graph and value semantics: module `i` imports modules
`i*4+1 .. i*4+4` (a 4-ary tree; 4 imports/module), and exports
`value_i = i + sum(imported values)`. The entry prints `value_0`, so a fresh
`n`-module corpus prints `n(n-1)/2`.

- **tiny** — the oracle's original module bodies: import lines plus one
  exported typed constant (~100 B/module). Parse/link overhead dominated.
- **realistic** — ~2.4 KB/module (min 2.3, max 2.6) of plausible TypeScript:
  an interface, string/object/array literals, four exported functions (one
  live in the value computation, three unused and tree-shakeable), typed
  helpers, template literals, spreads, comments.

Sizes: 1,000 and 10,000 modules (0.10 / 1.04 MB source for tiny,
2.39 / 24.24 MB for realistic).

**Runtime verification (hard gate):** a (bundler, corpus) pair is never timed
unless its output bundle, executed under `node`, prints the independently
computed expected value. Every timed cold run and every incremental rebuild
is re-executed and re-checked. Disagreement excludes the pair and records the
reason in `results.json` instead of a number. In the final run reported here,
**every pair passed** — all five bundlers produced runtime-identical results
on all four corpora.

### Definition of "cold"

- `node_modules` installed; tool binaries and corpus files warm in the OS page
  cache (one untimed warmup run per pair precedes the timed runs).
- No persistent bundler caches: before **every** run (warmup, each timed run,
  and the RSS run) the harness deletes, per tool:
  - diffpack: the output file (diffpack has no cache directory)
  - esbuild: the output file (no persistent cache)
  - rolldown 1.2.0: the output file (no persistent cache)
  - rspack: the output directory and `<corpus>/node_modules` (its default
    `node_modules/.cache` location; persistent cache is also not enabled in
    the config)
  - vite: the output directory, the explicit cache dir `<corpus>/.vite-cache`,
    and `<corpus>/node_modules`
- Fresh process per run. Wall time is measured around the whole process, so it
  includes process startup — the cost a user actually pays per invocation.
  diffpack and esbuild are native binaries; rolldown, rspack, and vite run
  under Node (measured on this machine: bare `node -e ""` is ~13.8 ms; Node +
  `import("rolldown")` is ~49.1 ms). This startup difference is real, but it
  does not change any ranking below by itself.
- Cold build wall time is the **median of 5 timed runs**.

### Matched flags

Minify OFF, sourcemaps OFF, tree shaking ON, target esnext, single ESM chunk.
Exact commands/configs are in `bench/run.mjs` and `bench/tools/`. Deviations a
tool cannot avoid:

- **diffpack** (`diffpack bundle <entry> <out>`): tree shaking is its
  always-on conservative scope-hoisted path and cannot be toggled; it keeps
  code a full tree shaker would drop (visible in output sizes below). Output
  is a single plain-script chunk without `export` statements (app-bundle
  semantics); the other tools emit ESM that re-exports the entry's exports.
- **esbuild** (`esbuild <entry> --bundle --format=esm --target=esnext`):
  fully matched.
- **rolldown 1.2.0** (API: `treeshake: true`, `format: "esm"`,
  `codeSplitting: false`, `minify: false`, `sourcemap: false`): fully matched.
- **rspack 2.1.4**: ESM output needs `experiments.outputModule`; closest
  target is `["node20", "es2022"]` (no literal "esnext"); TypeScript via
  `builtin:swc-loader`; `mode: "production"` (tree shaking + module
  concatenation) with `optimization.minimize: false`. Its output keeps
  webpack-style runtime helpers, so its sizes are structurally larger.
- **vite 8.1.5** (`build.lib`, `formats: ["es"]`, `minify: false`,
  `sourcemap: false`, `target: "esnext"`): fully matched in flags, but Vite 8
  uses its own bundled rolldown, which const-inlines during tree shaking — on
  the all-constant tiny profile its output collapses to ~0.2 KB. As with the
  minified-constant caveat in `ROLLDOWN_COMPARISON.md`, that size is a
  constant-folding artifact of the corpus, not a general result.

### Incremental rebuilds

The edit is the oracle's content edit: module `n/2`'s exported base value
changes (expected printed value shifts by exactly `n`), applied atomically
(temp file + rename, like an editor's atomic save). 2 warmup edits, then
**median of 5 timed edits**, alternating edited/original state, each rebuild
runtime-verified. What each tool's number includes differs by what its API
exposes — read the table with this in mind:

| Tool | Mechanism | Includes watch detection? |
| --- | --- | --- |
| diffpack | existing `diffpack watch` (same engine as `oracle/benchmark.mjs`); wall time from edit to its post-emit "rebuilt" line | **yes** (inotify + stdout pipe) |
| esbuild | in-process `context.rebuild()` (incremental cache), incl. output write | no (rebuild on demand) |
| rolldown | `watch` + `incrementalBuild: true` (experimental), rolldown's own `BUNDLE_END` `event.duration` | no |
| rspack | in-process `compiler.watch` (`aggregateTimeout: 0`), wall time from edit to done-callback after emit | yes |
| vite | **skipped** — `vite build` has no rebuild API; vite's incremental story is dev-server HMR (unbundled, on-demand transform), a different axis than batch rebuild time | — |

So diffpack's incremental numbers carry watch latency that esbuild's and
rolldown's do not; the comparison is tilted against diffpack, which makes the
wins below conservative.

### Other metrics

- **Peak RSS**: `/usr/bin/time -v` around one standalone cold-build process
  (caches cleared first), "Maximum resident set size".
- **Output bytes**: sum of all emitted files; gzip = zlib level 6 per file,
  summed.

## Results: real application (integration/tanstack-start-reference)

Production builds at each tool's production defaults (minification ON — this
section deliberately measures the real "build the app" command, unlike the
matched-flags synthetic section). Before each diffpack client run the whole
`.diffpack-output/` is deleted; before each ssr run its `server/` output is
deleted (the client route manifest is a required input of the ssr build and
is kept); before each vite run `.output/`, `.tanstack/`,
`node_modules/.vite`, `node_modules/.tmp`, and `node_modules/.cache` are
deleted. Medians of 5 runs, fresh process each:

| Case | Wall time | Peak RSS | Output |
| --- | ---: | ---: | ---: |
| `diffpack build-app . client` | 441 ms | 185 MB | 731.1 KB (38 files) |
| `diffpack build-app . ssr` | 444 ms | 186 MB | 715.0 KB (42 files) |
| diffpack client + ssr (sum of medians) | 885 ms | 186 MB peak | 1,446.1 KB |
| `vite build` (client + SSR + nitro in one command) | 860 ms | 392 MB | 1,465.5 KB (83 files) |
| `npm run build` (reference: `vite build && tsc --noEmit`) | 2,265 ms | 420 MB | — |

Honest reading: on the full app build, diffpack (client+ssr, 885 ms) and
`vite build` (860 ms) are a statistical tie — vite is ~3% faster and also
emits the nitro server wrapper. Diffpack's advantage here is memory (2.1x
lower peak RSS), not wall time. The reference `npm run build` is 2.6x slower
than diffpack's two builds, but that gap is mostly the `tsc --noEmit`
typecheck, which diffpack does not perform; `vite build` alone is the
apples-to-apples bundling comparison. (Fixture note: `npm ci` currently fails
there — the pinned lockfile is missing `lru-cache@11.5.2` — so the harness
requires `npm install --no-save`, which leaves the lockfile untouched.)

## Analysis

**Cold builds.** Diffpack is fastest on both tiny corpora (tie with esbuild
at 1,000: 16.2 vs 16.4 ms; 1.5x faster than esbuild/rolldown at 10,000) and
narrowly fastest on realistic-10,000 (259.9 vs rolldown's 263.4 ms — within
run-to-run noise, ranges overlap: 252-272 vs 261-267 ms). **It loses on
realistic-1,000: esbuild builds it 1.5x faster (23.7 vs 36.1 ms).** With
realistic module bodies, esbuild's parallel parse + full tree shake is
cheaper than diffpack's current frontend at small-to-medium graph sizes; the
gap closes and reverses at 10,000 modules. The "faster than every bundler on
cold builds" goal is therefore not yet met — the precise, supported statement
is: fastest or tied on 3 of 4 synthetic corpora, 1.5x behind esbuild on the
realistic 1,000-module corpus, and a wall-time tie with Vite on the real app.

**Incremental rebuilds.** Diffpack is fastest on every corpus: 2.9x faster
than the best competitor (rolldown incremental) at tiny-1,000 (3.07 vs 9 ms),
3.4x at tiny-10,000 (30.05 vs 101 ms), 2.6x at realistic-1,000 (6.58 vs
17 ms), 2.5x at realistic-10,000 (65.25 vs 166 ms) — and 3-10x faster than
esbuild/rspack. These diffpack numbers *include* inotify detection, output
write, and pipe latency; rolldown's exclude detection. Note the gap here
(~2.5-3.4x) is smaller than the ~25x reported in `ROLLDOWN_COMPARISON.md`,
which measures diffpack's in-process edit path (excluding process/watch
overhead) against rolldown's watch: at 10,000 modules diffpack's engine time
is ~8 ms while the end-to-end watch loop measured here is ~30-65 ms —
detection latency and rewriting the whole output file dominate. Both numbers
are real; they answer different questions.

**Memory.** Diffpack's peak RSS is the lowest in every single measurement:
4.3x below the next-best on tiny-1,000 (10 vs 43 MB), 3.9-5.7x on
realistic-10,000 (172 vs 667-973 MB), and 2.1x on the real app (186 vs
392 MB).

**Output size — a real loss.** On the realistic corpora diffpack's
conservative tree shaker keeps unused-but-referenced code the others drop:
its realistic-10,000 output is 2.7x larger raw (7.9 vs 2.9 MB) and 3.6x
larger gzipped (641 vs 178 KB) than esbuild's. On the tiny corpora it is
mid-pack (smallest raw, but rolldown gzips smaller). Anyone optimizing
shipped bytes would not choose diffpack's current output on realistic code.

## Turbopack

Turbopack is not in the numeric tables. It ships only as part of Next.js;
`next build --turbopack` necessarily measures Next's framework pipeline
(app-router compilation, prerendering, route generation) on top of bundling,
which is not apples-to-apples against bare bundler invocations on a shared
corpus. A fair comparison needs a pinned Next.js fixture benchmarked
Next-vs-Next (e.g. `next build --turbopack` vs `next build` with webpack, and
a diffpack build of the equivalent app). Listed as future work.

## Addendum 2026-07-21: transitive statement-level shaking landed

After the tables above were recorded, diffpack's per-module shake became
transitive (`shake_module_code` liveness fixpoint): non-exported pure helpers
referenced only by dead exports now fall with them, and all obviously-pure
top-level declarations (object/array/template literal initializers included)
are removal candidates. Spot re-measurement with the same corpus generator and
the same runtime-verification gate (bundle must execute and print the
independently computed value — both did, on every run):

| Corpus | diffpack output (was) | diffpack output (now) | esbuild output |
| --- | ---: | ---: | ---: |
| realistic-1k | 795,910 B | **182,463 B** | 355,577 B |
| realistic-10k | ~3.6 MB (2.7x esbuild) | **1,874,454 B** | 3,621,068 B |

Diffpack's realistic-corpus output went from ~2.7x larger than esbuild to
~1.9x **smaller**. Cold wall time on realistic-10k measured ~0.29 s (3 runs,
consistent with the table above). These are single-cell spot checks run with
`bench/gen.mjs` + direct CLI invocations, not a full harness re-run; the full
tables above still show the pre-change sizes. Correctness after the change:
the full conformance suite (40/48, unchanged), both reference apps' acceptance
and browser gates, and the incremental/memory thesis guards all pass.

## Addendum 2026-07-21 (later): cold wall time — the 4-thread cap and allocator contention

Profiling the realistic-1k cold build (90 ms CPU but only ~2.7 CPUs utilized on
32 cores) found three compounding serializers, now fixed: the frontend pool was
capped at 4 threads (`.min(4)` — removed); discovery had a fork-join barrier
per BFS wave (replaced by pipelined spawning with sorted-path interning, so
output bytes stay deterministic — verified byte-identical across runs); and the
tracking allocator updated shared atomic counters on EVERY allocation from all
threads (accounting is now off unless `bundle-scale-memory`/the guard suite
enables it, and the inner allocator is MiMalloc). Spot re-measurement, medians
via `perf stat -r 10` (same corpora, outputs runtime-verified):

| Corpus | diffpack cold (was) | diffpack cold (now) | best rival (above) |
| --- | ---: | ---: | ---: |
| realistic-1k | 36.1 ms | **13.9 ms** | esbuild 23.7 ms |
| realistic-10k | 259.9 ms | **132 ms** | rolldown 263.4 ms |

With this, diffpack's measured cold time leads every rival cell above (tiny
corpora were already led and only get faster from the same fixes). The full
tables above predate the change; a fresh harness run should replace them.

## Reproducing

```console
cargo build --release
cd bench && npm ci
node run.mjs                  # full matrix + real app
```

See `bench/README.md` for options (subset runs merge into
`bench/results/results.json`). The harness refuses to time any pair whose
output does not execute and agree with the expected value, and records the
exclusion reason instead.
