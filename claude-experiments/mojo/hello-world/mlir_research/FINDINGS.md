# Reverse-engineering Mojo's `.mojo_cache` MLIR bytecode

Live research notes. Treat anything here as provisional until validated by
differential compilation.

## What the cache actually stores

Path: `<env>/share/max/cache/.mojo_cache/mojo/transform/<version>-production/`

Each Mojo compilation deposits **two files** there, named by content hash
(base64-encoded hex):

- `<hash>` — a 19-byte-magic file beginning with `4d 4c ef 52` ("ML\xefR")
- `<hash>` (no extension) — a regular Mach-O / ELF object file (the AOT-compiled host code)

Other than `transform/`, no other stage directories appear no matter how you
compile. Mojo doesn't persist `lower/`, `codegen/`, etc. — those run in-memory.

## The "MLIR bytecode" file is actually a wrapper

The full file layout is:

```
[ MLIR bytecode (sections 0..6 + 8) ]
[ Modular diagnostics trailer       ]
[ 16-byte content hash              ]
```

Confirmed by differential compilation: `def main(): pass` vs
`def main(): var x: Int = 5` produced **byte-identical MLIR bytecode** (the
unused var was DCE'd at the transform stage), but the second file was 189
bytes larger because the trailer stored the warning message:

```
01 01 01 00 ... 01 6a 00 ...
loc("/Users/.../one_var.mojo":2:18)\0
assignment to 'x' was never used; assign to '_' instead?\0
... [16-byte hash]
```

This is why warnings replay on cached rebuilds — they're stored in the
trailer, not regenerated by the compiler.

## MLIR bytecode header

```
offset  bytes                                      meaning
   0    4d 4c ef 52                                magic "ML\xefR"
   4    [varint]                                   bytecode version (= 6 in MLIR23.0.0git)
   5    NUL-terminated string                      producer ("MLIR23.0.0git\0")
  19    sections begin
```

Producer string `MLIR23.0.0git` ⇒ Modular's MLIR fork at "trunk-ish" LLVM 23.

## Section table

Each section: `<id_byte> <length_varint> [optional align varint] <payload bytes>`.
The `id_byte` low 7 bits = section ID; high bit = "has alignment".

For our `tiny.mojo` test case (`def main(): var x: Int = 2 + 3; print(x)`):

```
 off  id  name              len   notes
 ----------------------------------------------------------------
  19   1  Dialect            76   reorganized vs upstream — see below
  97   3  Attribute        1067   interned attribute records
1167   2  AttrTypeOffsets  4784   offset table indexing into Attr+Type sections
5954   4  Type             7169   interned type records (the largest section)
13126  6  IR                  1   vestigial — reduced to a placeholder byte
13129  5  Resource            0   empty (no embedded resources)
13131  0  String           6239   interned strings (op names, paths, mangled symbols, ...)
19373  8  ?                 576   THE ACTUAL OPERATION TREE — undocumented section ID
```

**Discoveries vs upstream MLIR documented format:**

1. Section ID **8** isn't documented in MLIR's public bytecode spec, which
   lists 0–7. In upstream MLIR ID=6 is the IR/op section; here it's reduced
   to 1 byte and the op tree moved to ID=8. Likely a Modular-specific
   reorganization (their fork string is "MLIR23.0.0git" — trunk).

2. Section **1 ("Dialect")** is small enough (76 bytes for tiny, 110 for
   cpu_demo) that it's clearly only listing dialect names — but its
   indexing scheme doesn't decode cleanly with upstream MLIR's per-dialect
   layout. We get garbage indices when reading it as
   `[numDialects][nameIdx, hasVersion]+ [numOps][dialectIdx, nameIdx]+`.
   So the **section format is changed** vs. upstream too — not just the
   section ID assignment.

## Varint encoding (unchanged from upstream)

```
first byte    bytes total    value formula
0x..1         1              value = byte >> 1
0x..10        2              value = (LE_u16) >> 2
0x.100        3              value = (LE_u24) >> 3
...
0x10000000    8              value = (LE_u64) >> 8
0x00000000    9              value = next 8 bytes as LE_u64
```

(Number of total bytes = `count_trailing_zeros(first_byte) + 1`, capped at 9.)

## String section (id=0) — fully decoded

```
  numStrings    : varint
  lengths[N]    : varint × N   (front-loaded for varint compactness)
  strings[N]    : packed bytes at the END of the section,
                  in REVERSE order (read from end of payload backward).
```

Each string is NUL-terminated.

For `tiny.mojo` we recovered 129 strings, broken down as:

| Bucket | Examples |
|--------|----------|
| Op names | `func`, `return`, `load`, `store`, `add`, `cmp`, `cast`, `cast_to_builtin`, `cast_from_builtin`, `simd.and`, `simd.xor`, `simd.select`, `loop`, `if`, `try`, `try.raise`, `try.yield`, `break`, `continue`, `yield`, `external_call`, `create_closure`, `stack_allocation`, `stack_alloc.lifetime.start`, `stack_alloc.lifetime.end`, `pointer_to_index`, `pointer.bitcast`, `union.bitcast`, `string.address`, `atomic.rmw`, `fence`, `floordiv`, `aligned_free`, `offset` |

## Differential compilation results

### Test: empty function vs unused var
- `def main(): pass` — 2331 bytes total
- `def main(): var x: Int = 5` — 2522 bytes total

**MLIR bytecode portion: byte-identical.** Mojo DCE'd the unused var at the
`transform` stage. The 191-byte difference is entirely in the diagnostic
trailer (warning message + source location).

### Test: print(5) vs print(6) vs print(1234567890)

| Section | print5 | print6 | print(big) |
|---|---|---|---|
| Dialect | 76 | 76 | 76 |
| Attribute | 1067 | **1068** | 1068 |
| AttrTypeOffsets | 4784 | **4787** | **4791** |
| Type | 7169 | 7169 | 7169 |
| IR (id=8) | 576 | **579** | 579 |
| String | 6241 | 6241 | **6244** |

**Conclusions confirmed:**
- Integer literals → **Attribute** section (fixed-width record, +1 byte per new attr)
- Attribute index table → **AttrTypeOffsets** (+3 bytes per new attr)
- Op tree (id=8) references attributes by ordinal → +3 bytes per new attr ref
- IR section is **value-agnostic** (576 → 579 same for `6` as for `1234567890`)
- String section grows with print(big) but not print6 (+3 bytes — likely a digit-count metadata interning)

## OpTree (id=8) op-name references — confirmed direct string-index encoding

Major finding: **section 8 stores op opcodes as direct varint indices into
the String section (id=0), not as packed `(dialect_id, op_id)` like upstream
MLIR**. We confirmed this by running a heuristic scan over both `empty.mlirbc`
and `tiny.mlirbc`:

For `empty.mlirbc` (`def main(): pass`, 109-byte OpTree):
```
offset  byte    decoded op
   5    0x1f →  main           (function name as string ref)
  35    0x45 →  create_closure
  56    0x47 →  return
  59    0x49 →  func
```

For `tiny.mlirbc` (`def main(): var x = 2+3; print(x)`, 576-byte OpTree)
the scanner found 30 op-name references — the recurring `create_closure +
return` pair appears 14 times (matching the closure-elaboration shape of
`print` and its formatting helpers), plus `atomic.rmw`, `and`, `cmp`,
`union.bitcast`.

**Implication for parsing**: we don't actually need to crack the (still-opaque)
Dialect section (id=1) to identify which ops a program uses. Just read the
String section, then scan section 8 for every byte whose varint value indexes
a string we've classified as an op-name.

The Dialect section probably still encodes per-dialect metadata (versions,
op→ID dedup tables) but the actual IR walks ops by string index rather than
by dialect-op-id. This is a notable departure from upstream MLIR bytecode.

## Reverse-engineering plan (continued)

Now tractable since we have working op-name scans:

1. ~~Identify what bytes are op opcodes in section 8~~ — done; they're
   string-section indices.
2. **Map operand/result/region encoding within an op record.** Each op
   reference in the IR is followed by some fixed/variable structure: result
   types, operands, attributes, successors, regions. We need to figure out
   the per-op-record layout by:
   - Compiling minimal programs that force a single op with N operands
     (e.g., `pop.add(a, b)` vs `pop.add(a, b, c)` if Mojo allows variadic)
   - Diffing OpTree bytes between them.
3. **Crack the Attribute and Type sections.** Both use AttrTypeOffsets (id=2)
   as an index. Plan: vary one attribute at a time and watch which 1-byte
   offset entries change.
4. **Crack the Dialect section (id=1).** Less critical now that we can read
   ops directly, but still needed for full decode (per-dialect versions,
   custom attribute/type encoders).

## Attribute & Type sections are canonically sorted (not append-only)

Byte-level diff of `print(5)` vs `print(6)` (using `byte_diff.py` /
`difflib.SequenceMatcher`) shows ~500 single-byte "replace" edits scattered
through both the Attribute and AttrTypeOffsets sections, plus localized
inserts/deletes:

```
delete      277  Attribute               177      1     0  A=0f  B=
insert      425  Attribute               325      0     1  A=  B=0f
insert      918  Attribute               818      0     1  A=  B=0f
insert     1820  AttrTypeOffsets         650      0     3  A=  B=160315
delete     1929  AttrTypeOffsets         759      3     0  A=110115  B=
insert     2606  AttrTypeOffsets        1436      0     3  A=  B=151101
delete     7942  Type                   1985     11     0  A=a6020301054f00a6024f00  B=
...
```

This rules out any "appended new entry" interpretation. Implication: the
Attribute, Type, and AttrTypeOffsets sections are stored in a **canonical
order** (hash-sorted, content-keyed) for file determinism. Changing `5` to
`6`:

1. Removes one IntegerAttr record from its old sort-position
2. Inserts the new IntegerAttr at its new sort-position
3. Shifts every offset entry in AttrTypeOffsets that pointed past the
   shuffled region
4. Updates the OpTree's reference index from "old slot" to "new slot"

This is consistent with MLIR's general design — bytecode is supposed to be
canonical for build reproducibility. It also makes byte-level
reverse-engineering of the per-attribute-record encoding much harder
because we can't isolate "the bytes that encode 5" from "the bytes that
were shifted to make room for the canonical reordering."

**To decode individual attribute records you'd need to either:**
1. Pin down the canonical-sort key (likely a hash function, MD5/SHA1/BLAKE2
   of the textual MLIR form). With the key known, you can predict where any
   given attr record will sort.
2. Use a real MLIR reader linked against `libMLIR` with stub dialects
   registered. The C++ `BytecodeReader` skips dialect-specific bytes by
   calling each dialect's `readAttribute()` / `readType()` — so a stub that
   just reports "unknown attr of N bytes" would still let us walk the
   sections structurally.

## Modular trailer format (preliminary)

Tail bytes after the last MLIR section. Two structural variants observed:

**Empty (no diagnostics)** — `empty.mojo`, 32 bytes:
```
01 01 00 00 00 00 00 00 00 00 fb 08 00 00 00 00
00 00 a6 44 ad c1 18 f6 17 ca 8e 5d 8a a0 31 ef 23 d2
```

**With one cached warning** — `one_var_unused.mojo`, 223 bytes:
```
01 01 01 00 00 00 00 00 00 00 01 6a 00 00 00 00 00 00
loc("/Users/.../one_var.mojo":2:18)\0
8 00 00 00 00 00 00 00
assignment to 'x' was never used; assign to '_' instead?\0
00 00 00 00 00 00 00 fd 08 00 00 00 00 00 00
[16-byte content hash]
```

Looks like:
```
trailer:
  prefix      : 8-10 bytes (likely struct of flags + version + counts)
  diagnostics : array of (severity, source-location-string, message-string,
                more-fields), with 8-byte-aligned padding
  hash        : final 16 bytes (content checksum)
```

Last 16 bytes look like a content hash. Likely candidates: BLAKE3 (default
output is 32 bytes, but truncated 16-byte form is common), MD5, SHA-256
truncated. Couldn't pin down which without the source.

## Open questions

- Per-record encoding within Attribute and Type sections (blocked by
  canonical-sort issue — see above).
- Format of the Dialect section (id=1) in Modular's MLIR fork (only 76 bytes
  in `tiny.mojo`, but doesn't decode with the upstream MLIR per-dialect
  format — Modular reorganized it).
- The exact prefix bytes in the Modular trailer (severity flags? source-loc
  count?).
- Hash algorithm used in the trailer (16-byte trailer ⇒ truncated
  SHA-256/BLAKE2/BLAKE3? or full MD5?).
- Why section ID=6 ("IR" in upstream) is reduced to 1 byte rather than
  removed entirely. Likely a marker/version byte.

## Solution: Modular's own Python bindings parse their bytecode

Modular's `max` package ships compiled MLIR Python bindings:
- `max._mlir._mlir_libs._mlir.cpython-...so` — their MLIR Python API
- `max._core.cpython-...so` — Modular's own C++ extension

The trick: register their dialects via `graph.load_modular_dialects(registry)`
which sets up `kgen`/`pop`/`m`/`mosh`/`mo`/`rmo`/`hlcf`/`interp` parsers,
then call `ir.Module.parse(bytes)`. The Modular trailer must be stripped
first (the upstream parser tries to read it as another section and fails).

```python
from max._mlir import ir
from max._core import graph
reg = ir.DialectRegistry()
graph.load_modular_dialects(reg)
ctx = ir.Context()
ctx.append_dialect_registry(reg)
ctx.load_all_available_dialects()

raw = open(path, "rb").read()
mlir_bytes = raw[:trailer_offset]   # use parse_mlir_bc.find_trailer()
mod = ir.Module.parse(mlir_bytes, ctx)
print(str(mod))   # full textual MLIR
```

`pixi add max` includes these bindings (the basic `pixi add mojo` does not).

See `dump_mlir.py` for a complete working tool. Sample outputs are committed
to `samples/*.mlir` next to each `.mlirbc`.

## Sample sizes (post-decode)

| Source | bytecode | textual MLIR | notes |
|---|---|---|---|
| `def main(): pass` | 2.3 KB | 44 lines | minimum baseline; runtime init machinery |
| `var x: Int = 5` (unused) | 2.5 KB | 44 lines | identical MLIR to baseline (DCE confirmed) |
| `print(5) / print(6) / print(1234567890)` | 20 KB | 969 lines | print drags in 6+ monomorphized stdlib funcs |
| `for i in 1..10000: total += sqrt(i)` | 124 KB | 8851 lines | a real program |

The tooling we *thought* we needed (custom C++ shim, LLVM 23 build, byte-level
attribute decoders) turned out to be unnecessary — Modular ships exactly
what we needed in their Python package, just one `import` away.

## Original reverse-engineering value

The byte-level findings above remain useful for:
- Understanding the file format without depending on Modular's runtime.
- Stripping the trailer (the `parse_mlir_bc.find_trailer()` step is what
  unblocks `Module.parse`).
- Debugging cases where the bindings are unavailable (e.g. an environment
  with only `mojo` installed, no `max`).
- Building tools that work on bytecode without loading every Mojo dialect
  into a Python process.

The Modular trailer format remains undocumented elsewhere. Our notes on
its structure (8-10 byte prefix, length-prefixed diagnostic strings, 16-byte
hash at the end) are still the only public reference.
