Glaze — a staged style & shader language

A style is a function. Half of it runs now, on the CPU. The other half is compiled into a shader and runs every frame, on the GPU. You never decide which half is which — the language figures it out.

The one idea. Every value in a style has a binding time:

static  depends only on tokens, the element's variant, its data attrs, and inherited environment. → folded to a constant now.

dynamic  depends (even indirectly) on a per-frame input: time · hover · mouse · focus · press · size. → compiled to WGSL.

This is binding-time analysis + partial evaluation — the same machinery behind a partial evaluator. The novelty is that the stage boundary is literally the GPU upload. None of the languages that "almost became CSS" (DSSSL, PSL96, JSSS) were staged.

The mental model

A Glaze style is a pure function (variant, state, attrs, env) → { box, paint }. It produces two things: box properties that feed the layout engine, and a paint stack (fills, borders, shadows, and shader layers) drawn into whatever box the layout engine computed.

// a style is just a function of its inputs. `intent` is a static parameter.
style button(intent) {
  fill   intent == danger ? danger.solid : accent.solid   // static → folds
  pad    8px 12px                                          // box → layout
  radius radius.md                                         // paint only

  overlay shader {                                         // a glow on top
    let pulse = 0.5 + 0.5 * sin(time * 2)              // time → dynamic
    emit smoothstep(0, 1, hover) * pulse * vec4(1,1,1,0.25)
  }
  :focus { border focus.ring 2px }                         // discrete state plan
}

How it relates to layout

This is the part worth getting exactly right. Glaze does not own layout — Taffy (the same flexbox/grid engine bevy_ui uses) does. The element tree defines structure; each node flows through two resolvers:

Element treestructure + handlers (Rhai / subprocess)
→
Taffyconsumes box props → (x,y,w,h)
→
Glaze paintfills + shaders into the box
→
Composite+ transform, per-frame uniforms

A Glaze style straddles the two: some properties are layout-affecting (pad, gap, width/min/max, align) and get handed to Taffy; the rest are paint-only (fill, radius, border, shadow, shaders) and never move anything.

The staging constraint on layout = CSS's best idea, kept

Layout runs on the CPU. So layout-affecting properties may be static or discrete-state, but never continuous-dynamic. You cannot bind pad to time — that would re-run Taffy every frame. Glaze rejects it at compile time (loudly).

Continuous motion has two cheaper homes, and the compiler routes you to them automatically:

You want…It becomes…Cost
different space occupied (push siblings)box prop → Taffyrelayout — only on discrete state (an event)
smooth movement (translate/scale/rotate)transform @ compositeno relayout, GPU-cheap
smooth appearance (glow, fill, distortion)shader layerGPU, per-frame uniforms

This is exactly the browser's reflow-vs-composite distinction — and it falls out of the static/dynamic split for free, instead of being a rule you have to memorize. The honest verdict on layout itself: Glaze ties CSS (Taffy ≈ flexbox/grid). The win is everywhere paint and transform live.

Interactive 2 — what a shader{} block actually compiles to

The button below is a WebGL canvas running the exact shader the overlay shader{} above lowers to. time auto-animates; drag hover. Everything static (the gold, the radius) was folded to a constant before the shader ever ran.

// Glaze source
overlay shader {
  let pulse = 0.5 + 0.5*sin(time*2)
  emit smoothstep(0,1,hover)
       * pulse * vec4(1,1,1,0.25)
}
// generated WGSL (uniforms bound by name)
struct U { time:f32, hover:f32 }
@group(2)@binding(0) var<uniform> u:U;
fn frag()->vec4<f32>{
  let pulse = 0.5+0.5*sin(u.time*2.0);
  let a = smoothstep(0.,1.,u.hover)*pulse*0.25;
  return vec4(a,a,a,a); }

This feeds the existing DynamicMaterial path: the host parses struct U with naga, learns each field's offset, and binds time/hover by name. The compiler is the new part; the GPU plumbing already exists.

Scenarios — does programmatic actually beat CSS?

The honest question. Verdicts are color-coded, including the ones where a CSS-like setup wins.

1 · Hover glow / focus pulse / animated fills only glaze

Arbitrary shader math (SDF, noise, refraction, gradients) parameterized by continuous state, across hundreds of elements on the GPU. CSS gives you transition and @keyframes over a fixed property set — it cannot express a procedural fragment. This is the whole reason for the language; see Interactive 2.

Verdict: impossible in a CSS-like setup. The staged shader is the point.

2 · Data-driven color (heatmaps, value→color) glaze better

Map a number through a colormap, declaratively: fill colormap(attr.load, viridis). In CSS you precompute every color in JS and write inline styles; the styling lives in your app code. In Glaze it's in the style, and if attr.load is dynamic it folds onto the GPU. Scrub the field — these cells recolor through a real colormap (the same one Glaze would fold):

Verdict: Glaze better. CSS can do it only by leaking styling into application JS.

3 · Derived tokens & modular scales glaze better

font.base * pow(1.25, level), lighten(accent, 0.06), pick_readable(bg) for contrast. Full expressions over the token graph, folded at compile time. CSS has narrowed this gap with calc(), color-mix() and relative-color syntax — so it's a smaller win than it once was — but contrast-aware picks and arbitrary math still beat what CSS can do without escaping to JS.

Verdict: Glaze better, margin shrinking as CSS catches up.

4 · Conditional on data vs. structure it depends

Striping and first/last are structural: CSS :nth-child(even) nails them with zero ceremony, and Glaze has no selectors, so the script must pass an index attr. But "tint the row when attr.value > 80" is a data predicate — native in Glaze (fill value > 80 ? warn : ok), awkward in CSS (you set a class in JS).

serviceload

Verdict: tie — CSS wins structural predicates, Glaze wins data predicates.

5 · Contextual styling ("buttons in a toolbar are smaller") css-like better

This is what descendant selectors are for: .toolbar button { … } restyles by context without touching the buttons. Glaze deliberately dropped selectors (they fight the script-generates-the-tree model and bring specificity debugging), so context must be passed explicitly as a variant: button(size: toolbar). For deeply contextual design systems, CSS's model is genuinely more ergonomic here.

Verdict: CSS-like wins. An honest cost of dropping selectors.

6 · Plain reusable styling (a card, a button variant) tie

For "give me a reusable card with a shadow and rounded corners," a data schema or CSS classes are just as good and more familiar. Glaze's expressions and staging add nothing here — and this is most styling. The language only earns its keep once scenarios 1–3 show up in earnest.

Verdict: tie. Don't build a compiler for this case alone.

So: is it worth the programmability?

The programmatic, staged nature buys exactly three things a CSS-like setup cannot give us:

  1. Arbitrary GPU shaders as a style layer, authored in the same language, parameterized by live state — scenario 1. This is unreachable from CSS, full stop.
  2. Styling computed from data (colormaps, predicates, derived tokens) inside the style instead of leaking into app code — scenarios 2–4.
  3. One source that auto-splits CPU/GPU via binding-time analysis — Interactive 1. You write the look once; the compiler decides what's a constant and what's a uniform. Nothing CSS-shaped does this because nothing CSS-shaped is staged.

And the honest costs, restated: no selectors (scenario 5), no familiarity/ecosystem, and a real engineering risk that lives in the runtime (one DynamicMaterial per shading element → draw-call pressure), not the compiler. For ordinary styling (scenario 6) it's a tie, so the value is back-loaded onto the visual ambition.
